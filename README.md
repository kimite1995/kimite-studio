# KIMITE STUDIO — 관리 편의성 중심 안내 (2026.06 기준)

## 현재 상황 요약
- 사이트는 Next.js + Vercel 정적 배포로 되어 있습니다.
- 모든 작품 데이터(제목, 설명, 썸네일 경로, 영상 URL)는 `data/works.ts` (또는 JSON) 파일에 하드코딩되어 있습니다.
- 현재 `/admin` 페이지가 있어서 로그인 후 드래그로 순서 변경, 텍스트 수정, 파일 선택은 가능하지만, 실제 파일 저장과 데이터 반영은 여전히 수동 작업(파일 복사 + git push)이 필요합니다.

이 구조는 **개발 초기**에는 괜찮지만, **장기적인 '관리' 편의성** 관점에서는 좋지 않습니다.

## 관리(콘텐츠 업데이트) 편의성 관점에서 추천 순위

### 1위 추천: Headless CMS 도입 (가장 추천)
**Sanity.io** 또는 **TinaCMS**를 도입하는 것이 장기적으로 가장 편합니다.

**왜 이게 가장 편한가?**
- 별도의 관리자 화면(Studio)이 생김 (로그인만 하면 됨)
- 드래그 앤 드랍으로 작품 순서 변경 가능
- 썸네일/영상 직접 업로드 가능 (미디어 라이브러리 제공)
- 텍스트 수정 → 바로 저장 → 사이트에 반영 (git push 불필요하거나 최소화)
- 영상은 외부 호스팅(Vimeo, Bunny.net, Mux 등) URL만 저장하면 됨 → Git 용량 문제 없음
- 비개발자(당신)가 혼자서도 충분히 관리 가능

**Sanity.io vs TinaCMS 간단 비교**
- **Sanity.io** (현재 가장 추천)
  - 미디어 업로드 경험이 매우 좋음
  - 무료 플랜이 충분히 넉넉함
  - 한국어 지원도 괜찮음
  - 커스텀 스튜디오 만들기 비교적 쉬움
- **TinaCMS**
  - Git에 직접 저장 (코드와 콘텐츠가 함께 관리됨)
  - 시각 편집(사이트 보면서 수정)이 강점
  - Sanity보다는 미디어 경험이 약간 떨어질 수 있음

**필요한 작업 (대략)**
1. Sanity 계정 생성 + 프로젝트 생성
2. Next.js에 Sanity 연결 (이미 만들어진 컴포넌트 재사용 가능)
3. 작품 스키마 정의 (제목, 설명, 썸네일, videoUrl, category 등)
4. 기존 데이터 마이그레이션 (한 번만 하면 됨)
5. `/admin` 페이지는 Sanity Studio로 대체 (또는 병행)

이 방향으로 가면 앞으로는 **"Sanity Studio 접속 → 작품 수정/업로드 → Publish"** 만 하면 대부분 끝납니다.

### 2위: 현재 구조를 최대한 편하게 다듬기 (임시방편)
- `data/works.json`으로 변경 (TS보다 읽기/쓰기 쉬움)
- 관리자 페이지 UX 대폭 개선 (파일 업로드 시 자동 다운로드 + 안내 강화, 미리보기 강화)
- 영상은 무조건 외부 URL(Vimeo 추천)만 사용하도록 유도
- README에 "관리 매뉴얼"을 아주 자세히 작성

이 방법은 지금 당장 적용은 빠르지만, **장기적으로는 여전히 git push + 파일 관리**가 필요해서 피로도가 쌓입니다.

### 3위: 완전한 커스텀 백엔드 (비추천)
- Supabase + 관리자 페이지 + 파일 업로드 API
- 장점: 완전한 자유도
- 단점: 개발 비용이 크고, 유지보수도 필요. 당신이 코딩을 할 줄 모른다는 점에서 비효율적
